## 三、新增需求（本次补充）

| 需求 | 说明 | 归属 |
|------|------|------|
| **四级部门以下必须全部提交，四级以上允许单个提交** | 提交绩效审定时：四级部门以下必须全部提交才能到审定环节，四级以上部门允许单个提交审定 | 版本4考核活动管理 |

**理解**：
- 四级部门以下（如五级、六级部门）：必须该四级部门下所有下级人员都提交了绩效，才能进入审定
- 四级以上（如集团、部门级别）：允许单个提交，提交即可进入审定

## 七、提交阈值规则引擎（版本4）

### 需求来源

原始需求："四级部门以下必须全部提交，四级以上允许单个提交" + 按区域提交场景

### 需求本质

把"谁必须全部提交"变成可配置的阈值规则，而非写死代码。

### 分组维度（可组合、可配置）

"全部"的概念支持以下几种分组维度，每种可以独立配置是否启用：

| 分组维度 | 说明 | "全部"的范围 |
|----------|------|------------|
| **按部门层级** | 四级以下 / 四级以上 | 行政层级分组（组织架构字段，绩效系统调用API获取） |
| **按区域** | 欧洲区、美洲区等 | 同区域的员工（员工region字段，独立于部门） |
| **按考核组** | 百分制组、五分制组 | 同考核组的员工 |
| **按部门** | 同部门全员 | 同部门的员工 |

维度可单独启用或组合启用：
- 启用区域 + 启用部门层级 → 两个维度都需满足才算"全部"
- 只启用部门层级 → 只按部门层级判断

### 两种提交模式

| 模式 | 触发条件 | 效果 |
|------|----------|------|
| **全员提交** | 分组内所有人都提交了 | 解锁下一环节 |
| **单人通过** | 分组内任意1人提交 | 直接进入下一环节 |

### 配置示例

| 场景 | 配置方式 |
|------|---------|
| 四级以下必须全部提交，四级以上允许单个提交 | 启用"按部门层级"维度 + 四级以下100% / 四级以上1人 |
| 欧洲区全部提交后才能提交 | 启用"按区域"维度 + 区域内100% |
| 同一考核组必须一起提交 | 启用"按考核组"维度 + 组内100% |
| 五级部门以下全部提交 | 启用"按部门层级"维度 + 五级以下100% |

### "全部"范围的锁定时机

```
考核活动初始化
    ↓
根据考核方案配置，确定参与人员
    ↓
按分组维度生成分组名单（活动生命周期内固定，除非管理员手动调整）
    ↓
流程流转过程中，持续追踪每个分组内已提交的人数
    ↓
达到配置的阈值（100%或单人） → 解锁下一环节
```

### 数据流转

```
组织架构
    ├── 员工 → 部门层级字段（N级，行政层级）
    ├── 员工 → 区域字段（BU-区域，2级）
    └── 员工 → 汇报关系（按汇报线逐级审批）

绩效系统
    ├── 调用组织架构API获取员工维度信息（部门层级、区域）
    ├── 考核活动初始化时锁定分组名单
    ├── 流程引擎汇报每个环节完成状态
    └── 绩效系统判断是否达到阈值，决定是否解锁下一环节

流程引擎
    └── 只负责环节内的流转，不判断"全部"的阈值
```

### 两种业务场景的处理路径

| 场景 | 处理方式 |
|------|---------|
| **按部门层级** | 绩效系统查组织架构 → 获取每个参与人的行政层级 → 审定环节时按部门分组 → 四级以下：组内100%提交才解锁 → 四级以上：单人提交即解锁 |
| **按汇报线逐级** | 流程引擎按汇报关系逐级取处理人 → 每级完成后进入下一级 → 最上级完成才算全部完成 → 绩效系统监控整条汇报链的状态 |

### 关键约束

1. **按部门层级的"全部"**：锚定考核活动初始化时确定的参与人员名单，活动生命周期内固定
2. **按区域分组**：区域是独立字段，与部门平行，目前2级（BU-区域）
3. **部门层级与汇报线的关系**：大部分场景汇报线和部门层级一致；存在跳级汇报的小概率场景，方案优先覆盖大部分场景
4. **流程引擎职责**：只负责环节内节点流转，不判断"全部"阈值，阈值判断由绩效系统控制
