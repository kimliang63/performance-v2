# 考勤机组织人员推送 - 实时推送优化需求

## 一、现状分析

### 1.1 当前架构

```
OEHR/OTWS（业务系统）
       ↓ 定时拉取（最快4小时一次）
DW（数据仓库）
       ↓
推送服务（中控）
       ↓
考勤中控
```

### 1.2 当前存在两大核心问题

#### 问题一：历史数据迁移难题

**背景**：每个环境按区域/部门逐步迁移上线

**现状问题**：
- 每次迁移都需要研发手动记录一次数据，然后备份
- 推送时需要比对判断这些数据是否在**上线之前就存在**于系统里
- 上线前的存量数据不应该被重复推送（否则会造成数据重复或状态错乱）

**本质**：缺乏有效的数据溯源机制，无法区分"存量数据"和"增量数据"

---

#### 问题二：推送实时性差

**现状**：
- DW 定时拉取业务数据，最快只能 4 小时拉一次
- 实际数据更新频率至少 4 小时起步
- 做不到实时或者类实时推送

**影响**：
- 员工入职后需要等 4 小时才能在考勤机上出现
- 组织变动后需要等 4 小时才能同步
- 业务部门投诉响应慢

---

## 二、优化目标

**核心目标**：扔掉 DW，实现实时推送

**优化后架构**：
```
OEHR/OTWS（业务系统）
       ↓ 业务事件触发（实时）
推送服务（中控）
       ↓
考勤中控
```

**预期效果**：
- 业务数据变更时，立即触发推送
- 数据更新延迟从 4 小时降为秒级
- 消除 DW 这个中间层，简化架构

---

## 三、需要改动的事项

### 3.1 触发机制重构

| 维度 | 当前 | 优化后 |
|------|------|--------|
| 触发方式 | 定时拉取（DW -> 推送服务） | 事件驱动（OEHR/OTWS -> 推送服务） |
| 触发时机 | 每4小时一次 | 业务变更时实时 |
| 数据来源 | DW 主动拉取 | OEHR/OTWS 主动推送 |
| 实时性 | 差（4小时+） | 好（秒级） |

**需要改动**：
- OEHR/OTWS 侧：业务变更时调用推送服务接口
- 推送服务侧：接收业务系统的主动推送请求

---

### 3.2 历史数据处理机制

**核心问题**：如何判断数据是否在迁移前就存在？

**优化思路**：建立数据生命周期标记机制

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| 方案A：迁移时间戳 | 记录每个环境的迁移时间点，迁移时间前创建的数据标记为存量 | 简单，但需要手动管理迁移时间 |
| 方案B：首次同步标记 | 数据首次出现在系统时标记，后续推送可识别 | 更精确，需要改造数据结构 |
| 方案C：双向比对 | 保留历史备份数据，新推送时与备份比对 | 当前方案，研发手动操作繁琐 |

**建议**：采用方案A + 方案B 结合

---

### 3.3 数据去重逻辑调整

**当前逻辑**（基于备份比对）：
```
增量数据 vs 历史备份 → 判断是否首次出现 → 决定是否推送
```

**优化后逻辑**（基于数据标记）：
```
新推送数据 → 检查数据标记 →
  - 存量数据标记：跳过推送或标记为不处理
  - 增量数据标记：正常推送
```

---

### 3.4 推送失败重试机制

**当前**：
- 失败后30秒重试一次，最多重试2次
- 重试后仍失败则记入失败数据

**优化后需要考虑**：
- 实时推送场景下，失败重试机制是否需要调整
- 是否需要增加失败告警机制
- 推送服务的幂等性保证

---

## 四、可能涉及的新功能

### 4.1 实时推送通道

**功能描述**：
- OEHR/OTWS 业务数据变更时，通过 API 主动调用推送服务
- 不再依赖 DW 定时拉取

**可能涉及**：
- 推送服务 API 接口开发（接收外部系统推送）
- OEHR/OTWS 侧消息通知机制开发
- 消息队列（如果调用方式改为异步）

---

### 4.2 数据生命周期管理

**功能描述**：
- 对每一笔推送数据标记生命周期状态
- 区分：存量数据、增量数据、已删除数据

**可能涉及**：
- 数据标记字段新增（数据来源、时间戳、状态）
- 迁移时间点配置功能
- 存量数据查询和管理

---

### 4.3 迁移过渡期支持

**功能描述**：
- 支持按环境、按区域逐步迁移上线
- 过渡期内双轨并行（ old 逻辑 + new 逻辑）
- 平滑切换

**可能涉及**：
- 环境级别的推送开关
- 灰度发布机制
- 切换日志和回滚机制

---

### 4.4 实时监控大屏

**功能描述**：
- 实时展示推送数据量、推送延迟、失败率
- 业务数据变更后可即时看到推送结果

**可能涉及**：
- 推送数据统计看板
- 告警规则配置（失败率超阈值告警）
- 推送日志实时查询

---

### 4.5 推送幂等性保证

**功能描述**：
- 同样的数据重复推送不会造成数据重复或状态错乱
- 支持消息去重

**可能涉及**：
- 推送记录去重逻辑
- 唯一标识符设计
- 状态一致性保证

---

## 五、需要进一步确认的问题

1. **OEHR/OTWS 调用推送服务的具体方式**：
   - 同步调用还是异步消息队列？
   - 接口协议和签名机制？

2. **迁移过渡期的业务连续性**：
   - 是否需要支持新旧两套逻辑并行？
   - 过渡期预计多长？

3. **数据标记的改造范围**：
   - 推送服务数据结构改造是否涉及考勤中控侧？
   - 历史数据如何处理？

4. **失败告警和监控要求**：
   - 实时推送场景下，失败率要求是多少？
   - 是否需要 7x24 监控？

---

## 六、初步评估

| 维度 | 评估 |
|------|------|
| 架构复杂度 | 降低（去掉 DW 这一层） |
| 实时性 | 大幅提升（从4小时到秒级） |
| 开发工作量 | 中等（需要新增实时推送通道、数据标记机制） |
| 迁移风险 | 需要关注（过渡期双轨并行） |
| 运维复杂度 | 需要新增监控和告警 |

---

## 七、后续工作建议

1. **第一步**：与 OEHR/OTWS 团队对接，确认业务系统主动推送的技术方案
2. **第二步**：设计数据生命周期标记机制，明确存量/增量数据区分方案
3. **第三步**：制定过渡期方案，确保历史环境迁移平滑
4. **第四步**：开发实时推送通道，同步改造推送服务
5. **第五步**：上线后监控调优，完善告警机制
