# 考勤机组织人员推送 - 实时推送优化详细方案

## 一、需求概述

### 1.1 优化目标

- **核心目标**：取消 DW 中间层，实现业务数据变更实时推送
- **优化效果**：数据更新延迟从 4 小时降至秒级

### 1.2 优化后架构

```
【当前】
OEHR/OTWS  ──(4小时轮询)──►  DW  ──(拉取)──►  推送服务  ──(推送)──►  考勤中控

【优化后】
OEHR/OTWS  ──(实时触发)──►  推送服务  ──(推送)──►  考勤中控
                                    ↑
                                  DW 删除
```

## 三、existing_data 表核心机制

### 3.1 existing_data 表的作用

**existing_data 表是判断数据是否已在考勤中控存在的核心依据。**

| 表字段 | 说明 |
|--------|------|
| id | 主键 |
| source_system | 来源系统：OEHR / OTWS_US / OTWS_DE |
| data_type | 数据类型：ORG / PERSON |
| data_key | 数据标识：orgCode 或 empNO |
| data_content | 完整数据内容（JSON） |
| exist_status | 存在状态：EXISTS / DELETED |
| migrate_area | 迁移区域（用于逐步迁移） |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 3.2 existing_data 表维护逻辑（不变）

| 场景 | 处理方式 |
|------|----------|
| **迁移时** | 人工整理数据 → 批量导入考勤机后台 → 记录到 existing_data 表 |
| **日常推送-新增成功** | 将数据写入 existing_data 表 |
| **日常推送-更新成功** | 更新 existing_data 表中的记录 |
| **日常推送-报错** | 记录失败原因（考勤中控里可能已被删除） |
| **考勤机后台人工删除** | 推送服务无法知道，推送时报错提示无法更新，保留此逻辑 |

### 3.3 逐步迁移的处理

每个区域迁移时，只更新该区域的记录：

```
migrate_area = 'US'  →  美区数据
migrate_area = 'DE'  →  欧区数据
migrate_area = 'GROUP' → 集团数据
```

### 4.2 改造功能（原有逻辑调整）

| 功能 | 当前逻辑 | 改造后逻辑 | 改动点 |
|------|----------|------------|--------|
| **触发机制** | DW 定时拉取（4小时） | 接收 OEHR/OTWS 主动调用 | 删除 DW 拉取逻辑，新增 API 接收器 |
| **数据来源** | 从 DW 查询数据 | 接收 OEHR/OTWS 推送的数据 | 数据获取方式改变 |

### 4.3 新增功能（从零开发）

#### 4.3.1 实时推送接收接口

**接口设计**：

```
POST /api/push/receive
Content-Type: application/json

{
  "sourceSystem": "OEHR",           // 来源系统：OEHR / OTWS_US / OTWS_DE
  "eventType": "EMPLOYEE_ADD",     // 事件类型（见下表）
  "dataType": "PERSON",            // 数据类型：ORG / PERSON
  "data": {                        // 业务数据（现有字段，不需要新增）
    "empNO": "E001",
    "empName": "张三",
    "orgCode": "815744",
    ...
  },
  "eventTime": "2025-01-15T10:30:00Z",  // 事件发生时间
  "traceId": "uuid-xxx"            // 追踪ID，用于幂等
}
```

**事件类型**：

| 事件类型 | 说明 | 对应推送类型 |
|----------|------|--------------|
| EMPLOYEE_ADD | 员工入职/新增 | 新增 |
| EMPLOYEE_TRANSFER | 员工调动 | 更新 |
| EMPLOYEE_LEAVE | 员工离职 | 离职 |
| EMPLOYEE_REHIRE | 员工离职恢复 | 离职恢复 |
| EMPLOYEE_DISABLE | 标记不可用 | 离职 |
| EMPLOYEE_ENABLE | 恢复可用 | 离职恢复 |
| ORG_ADD | 组织新增 | 新增 |
| ORG_UPDATE | 组织更新 | 更新 |
| ORG_DISABLE | 组织停用 | 删除 |

**接口职责**：
1. 接收数据并做基础校验（必填字段、格式）
2. 查询 existing_data 表，判断推送策略
3. 调用原有推送逻辑
4. 返回接收结果

#### 4.3.2 幂等性保证

**问题**：OEHR/OTWS 可能因为重试或延迟，重复发送同一笔数据

**解决方案**：基于 traceId 做去重

```
接收数据 → 检查 traceId 是否已处理 →
  已处理：返回成功，不重复推送
  未处理：正常处理并记录
```

## 五、推送判断逻辑（复用 existing_data 表）

### 5.1 判断流程

```
OEHR/OTWS 推送数据
       │
       ▼
查询 existing_data 表
       │
       ├── 不存在 ──→ 新增 ──→ 写入 existing_data 表
       │
       ├── 存在且 status=EXISTS ──→ 更新 ──→ 更新 existing_data 表
       │
       └── 存在且 status=DELETED ──→ 推送报错（考勤中控已删除）
```

### 5.2 与原 PRD 逻辑对比

| 判断维度 | 原 DW 方案 | 改造后实时推送 |
|----------|------------|----------------|
| **已存在判断** | existing_data 表 | existing_data 表（不变） |
| **新增判断** | 不在 existing_data 表 | 不在 existing_data 表（不变） |
| **更新判断** | 已在 existing_data 表 | 已在 existing_data 表（不变） |
| **删除处理** | 考勤机删了，推送报错 | 考勤机删了，推送报错（不变） |

**结论**：existing_data 表的维护逻辑和判断逻辑完全不变。

## 六、与迁移流程的关系

### 6.1 迁移时（不变）

```
1. 人工整理需要迁移的数据
2. 通过导入功能批量更新考勤机后台的组织节点
3. 将迁移区域的数据记录到 existing_data 表
   - migrate_area = 对应区域
   - exist_status = EXISTS
4. 后续推送时，查询 existing_data 表判断
```

### 6.2 日常推送（改造触发方式）

```
1. OEHR/OTWS 业务变更
2. 调用推送接口（POST /api/push/receive）
3. 推送服务查询 existing_data 表判断推送策略
4. 执行推送逻辑
5. 更新 existing_data 表
```

### 6.3 逐步迁移的支持

- 按区域逐步迁移，每个区域独立维护 migrate_area
- 新增推送不受 migrate_area 限制
- 迁移时只需更新该区域的 existing_data 记录

## 九、迁移策略

### 9.1 迁移原则

1. **按环境迁移**：测试环境 → 预生产环境 → 生产环境
2. **按区域迁移**：美区 → 欧区
3. **平滑切换**：新逻辑启用后，旧 DW 拉取逻辑停用

### 9.2 迁移步骤

| Phase | 步骤 | 工作内容 | 负责方 |
|-------|------|----------|--------|
| 1 | 开发 | 新增 API 接收接口 + 幂等校验 | 研发 |
| 2 | 开发 | 删除 DW 拉取逻辑，改造为接收模式 | 研发 |
| 3 | OEHR/OTWS | 接入推送接口 | 研发 + OEHR/OTWS |
| 4 | 测试 | 功能测试 + 联调测试 | 测试 |
| 5 | 测试 | 测试环境验证 | 测试 |
| 6 | 上线 | 生产环境切换，停用 DW | 研发 + 运维 |

### 9.3 回滚方案

| 触发条件 | 回滚动作 |
|----------|----------|
| 新逻辑推送失败率异常 | 关闭实时推送，恢复 DW 拉取（保留） |
