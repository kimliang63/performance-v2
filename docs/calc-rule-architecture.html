一、时序与分层（核心约束）
核心约束：公式是 COE 先维护的，表单是员工后填写的。配置公式时根本没有表单数据。
因此：公式编辑器的业务字段注册来源是配置层（指标→衡量标准→grades），不是表单。

  • 系统按预配公式计算得分
  • 结果写入明细表
执行层只提供输入值，不参与公式定义
二、衡量标准 / 指标库 / 计算规则 的关系
核心原则：衡量标准只存参考值数据（grades），不含计算逻辑。计算规则只定义公式，不含参考值。指标库把两者绑定：这个指标用哪些参考值 + 用哪个公式算出得分。
📏 指标库（Indicator）— 绑定者

定义"考核什么"，同时绑定参考值和公式

  • 指标名称：季度营收达成率
  • 指标编码：KPI_001
  • 指标分类：业务目标/团队管理/能力/价值观
  • 计量单位：万元、次、%、分
  • FK → 衡量标准：参考值数据
  • FK → 计算规则：计算公式
→
FK
📐 衡量标准（MeasureStandard）— 参考值

定义"参考值"，纯数据，不含计算逻辑

  • 标准名称：营收类参考值
  • type: 百分比 / 分值 / 等级（仅分类标签）
  • grades 数组：
  • [{name, min, max, desc}]
  • 如：挑战值120, 目标值100, 保底值80
  • ⚠️ 只存数据，不定义怎么算
←
FK
🧮 计算规则（CalcRule）— 公式

定义"怎么算"，公式决定参考值→得分的映射

  • 规则名称：梯度插值计算
  • 公式表达式：公式编辑器配置
  • 任意计算逻辑：
  • 梯度插值、封顶、线性、是否达成...
  • 同一公式可复用到不同指标
三、公式编辑器字段注册来源
公式编辑器的字段来自配置层，不来自表单。
COE 打开公式编辑器 → 选择指标 → 系统读取该指标的 FK→measure_standard → 拿到 grades → 注册为公式变量。
这时候表单数据还不存在。
注册链路
// COE 打开公式编辑器，选择"指标A：季度营收达成率" indicator = getIndicator("KPI_001") measure_std = indicator.measure_standard // FK → 衡量标准 grades = measure_std.grades // 拿到参考值数组 // grades 注册为公式变量： // grades[0].max → challenge_value (120) // grades[1].max → target_value (100) // grades[2].max → baseline_value (80) // grades[0].name → grade_name_0 ("卓越") // grades[0].min → grade_min_0 (100) // ... // 公式编辑器左侧字段面板 = grades 中的所有字段 // COE 用这些字段构建公式，如： // IF(actual_value > challenge_value, 120, ...) // 公式保存后，员工填表单时： // actual_value = 员工填的完成值（表单数据） // challenge_value = grades[0].max（配置层数据，早就配好了） // 系统执行公式 → 算出得分
不同指标 → 不同 grades → 不同字段

四、两层级公式设计
层级1：单个指标得分
配置什么每个指标的评分公式
字段来源该指标的衡量标准 grades + 表单字段
输入grades 参考值 + 员工填的完成值
输出单个指标得分
配置时机COE 配指标时（配置层，先于表单）
// 指标A的公式 IF(actual_value > challenge_value, 120, IF(actual_value >= target_value, 100 + (actual_value-target_value)/ (challenge_value-target_value)*20, 0))
层级2：总分汇总
配置什么总分计算公式（加权求和/自定义）
字段来源各指标的得分结果 + 权重 + 加减分
输入indicator_score_A, weight_A, bonus, penalty
输出总分
配置时机COE 配方案环节时（配置层）
// 总分公式 ROUND( indicator_score_A * weight_A / 100 + indicator_score_B * weight_B / 100 + indicator_score_C * weight_C / 100 + bonus - penalty , 1)
数据流
配置层（COE 操作，先于表单） ┌─────────────────────────────────────────────────────────────────────┐ │ 衡量标准 grades ──FK──→ 指标 ──FK──→ 计算规则（公式） │ │ [120,100,80] KPI_001 IF(actual>120, 120, ...) │ └─────────────────────────────────────────────────────────────────────┘ ↓ 方案引用 执行层（员工操作，表单填写后） ┌─────────────────────────────────────────────────────────────────────┐ │ 员工填表单 → actual_value=115 │ │ ↓ │ │ 系统执行公式: IF(115>120, 120, IF(115>=100, 100+(115-100)/(120-100)*20, 0)) │ │ ↓ │ │ 输出: indicator_score = 115.0 │ └─────────────────────────────────────────────────────────────────────┘ ↓ 层级1输出作为层级2输入 ┌─────────────────────────────────────────────────────────────────────┐ │ 总分 = 115.0×60/100 + 99.5×25/100 + 80×15/100 + 5 - 0 │ │ = 110.9分 │ └─────────────────────────────────────────────────────────────────────┘
五、数据模型
📐 measure_standard
  • PK id bigint
  • name varchar(200)
  • dept varchar(100)
  • type enum(百分比/分值/等级)
  • grades JSON ← 只存参考值
grades=[{name,min,max,desc}]
📏 indicator
  • PK id bigint
  • code varchar(50) UNIQUE
  • name varchar(200)
  • category varchar(50)
  • unit varchar(20)
  • FK measure_std_id → 衡量标准
  • FK calc_rule_id → 计算规则 ← 绑定公式
🧮 calc_rule
  • PK id bigint
  • name varchar(200)
  • formula text ← 只存公式
  • formula_vars JSON
  • status enum(enabled/disabled)

-- 指标库 → 衡量标准（多对一：多个指标可共用同一套参考值） indicator.measure_std_id → measure_standard.id -- 指标库 → 计算规则（多对一：多个指标可共用同一套公式） indicator.calc_rule_id → calc_rule.id -- 公式编辑器取字段（配置层，先于表单） -- COE选指标 → 读indicator.FK→measure_standard → 取grades → 注册为公式变量 -- 执行时（表单填写后） -- 员工填actual_value → 系统执行formula(challenge_value=grades[0].max, actual_value=115) -- 方案环节 → 计算规则（方案选择用哪套总分公式） perf_stage_config(stage='eval').calc_rule_id → calc_rule.id
