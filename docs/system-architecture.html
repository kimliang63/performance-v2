关键数据流
正向流转（标准路径）
  • HRBP 发起活动 → 从方案 copy 全部配置 → 生成活动快照
  • 活动开启环节 → 为每个员工创建流程实例 → 生成待办
  • 员工/上级处理待办 → 数据写入明细表 → 流转下一节点
  • 环节所有实例完成 → 环节状态=completed → 触发下一环节
  • 全部环节完成 → 活动状态=completed → 数据汇总到结果表
数据级联（表单间）
  • 目标提交 → 写入 goal_detail，状态=pending（审批中），数据可见
  • 目标审批通过 → 同一条记录状态=approved（生效），不复制
  • 目标驳回 → 同一条记录状态=rejected（未生效），员工可修改后重新提交
  • 考核：每个节点人提交 = 新增一行 eval_detail，驳回重提 = 再新增一行，旧行标记作废
  • 审定：每个节点人提交 = 新增一行 ratification_detail，旧行作废
  • 面谈 completed → 写入 interview_detail，确认表实时引用面谈数据
  • 员工确认 completed → 绩效结果锁定，不可修改

①
目标制定环节
员工填写目标 → 审批锁定 → 数据写入考核表
待办处理逻辑
  • 员工节点：填写目标名称/权重/衡量标准/目标值/挑战值/保底值，保存草稿或提交
  • 提交校验：目标数≥2、总权重=100%、必填字段完整
  • 间接上级节点：全部只读查看，同意/不同意/驳回/转交
  • 直接上级节点：全部只读查看，同意/不同意/驳回/转交
  • 空审批人：流程引擎自动跳过该节点
提交后数据处理
  • 保存草稿：数据写入 goal_detail，status=draft，不触发流程，不锁定
  • 提交：同一条记录 status=pending，数据可见，触发审批流程
  • 审批通过：同一条记录 status=approved（生效），目标数据锁定不可修改
  • 驳回：同一条记录 status=rejected（未生效），员工可修改后重新提交
  • 重新提交：同一条记录 status 回到 pending，数据更新，审批记录追加
  • 转交：当前待办转给他人，原处理人记录保留
  • 关键点：目标数据始终是一条记录，通过 status 区分审批状态，不复制不快照
②
绩效考核环节
自评 → 上级评分 → 计算总评价结果 → 数据写入审定表
待办处理逻辑
  • 自评节点：员工填写完成值 + 各指标评价结果 + 评语（目标数据只读引用自目标表）
  • 间接上级节点：查看完成值(只读) + 选择各指标评价 + 评语
  • 直接上级节点：查看+评分+填写总评价结果(必填)+总评语
  • 批量模式：同部门员工列表对比评分 + 批量导入
  • 无驳回：绩效考核流程仅提交，不支持驳回
提交后数据处理
  • 自评提交：新增一行 eval_detail（submitter_role=self, status=active）
  • 间接上级提交：新增一行 eval_detail（submitter_role=indirect, status=active）
  • 直接上级提交：新增一行 eval_detail（submitter_role=direct, status=active）
  • → 计算总评价结果（按节点权重加权 or 取最后填写值）
  • → 总评价结果+总分写入直接上级这行
  • → 触发结果审定环节开启
  • 驳回重提：间接上级驳回 → 员工修改 → 新增一行自评数据（status=active），旧行标记 voided
  • 批量提交：逐条处理，成功则提交，失败返回错误明细
  • 查询时：取该员工 status=active 的最新行 = 当前有效数据；status=voided 的行 = 历史记录可回溯
③
结果审定环节
逐级评级 → 强制分布校验 → HRBP审批（远超预期）→ 数据写入确认表
待办处理逻辑
  • 直接上级节点：查看考核结果(只读)+调整等级(必填)+查看强制分布实时统计
  • 间接上级节点：同上，查看+确认或调整等级
  • 逐级上级节点：按汇报线/组织架构逐级流转，到停止人员结束
  • HRBP节点：仅当调整后等级=A(远超预期)时触发，全部只读，同意/不同意
  • 强制分布校验：弱控→提示可提交；强控→禁止提交
提交后数据处理
  • 每次调整等级：实时更新强制分布计数（原子操作，防并发）
  • 直接上级提交：新增一行 ratification_detail（submitter_role=direct, status=active）
  • 间接上级提交：新增一行（submitter_role=indirect, status=active）
  • 逐级上级提交+到达停止人员：新增一行 + 审定完成 → 触发面谈+确认环节
  • HRBP驳回：当前行作废 → 回到直接上级 → 直接上级重新调整 → 新增一行
  • 跳过审定的员工：使用方案中配置的固定等级，纳入分布分母
  • 查询时：取 status=active 的最新行 = 当前审定结果；voided 行 = 历史回溯
⑤
绩效面谈环节
员工填写面谈记录 → 上级审批 → 数据写入面谈表+确认表引用
待办处理逻辑
  • 员工填写节点：面谈类型/亮点(≥3项)/不足(≥3项)/下阶段规划/需求(≥1项)/附件
  • 直接上级审批节点：查看面谈内容(只读) + 填写上级评语(必填) + 同意/驳回
  • 驳回：回到员工填写节点，员工修改后重新提交
提交后数据处理
  • 员工提交：面谈数据写入 interview_detail 表，状态=pending_leader_review
  • 上级同意（completed）：面谈数据锁定 → 写入 interview_result 表
  • 确认表引用：确认表单实时读取面谈数据（BR-023 直接引用不复制）
  • 上级驳回：状态回到 pending_employee_fill，员工可重新编辑
④
结果确认环节
员工查看全部结果 → 确认/不同意 → HRBP知悉（仅不同意时）
待办处理逻辑
  • 员工确认节点：全部只读展示（考核结果+面谈结果+审定结果），点击同意/不同意
  • HRBP知悉节点：仅员工"不同意"时触发，全部只读，点击"知悉"→流程结束
  • 超时处理：超过配置天数未操作 → 系统自动确认为"同意"（BR-005）
  • 数据全部只读：确认环节不可修改任何绩效数据
提交后数据处理
  • 员工同意：绩效结果锁定，状态=completed → 写入 final_result 表
  • 员工不同意：触发 HRBP 知悉 → HRBP 知悉后流程结束 → 结果仍锁定（线下处理）
  • 超时自动确认：系统以"同意"处理，记录自动确认标记
  • 结果不可逆：确认完成后终态不可逆，如需修正需发起新流程实例

计算引擎规格
1. 指标得分计算
定量指标得分
  • 公式：得分 = (实际值 - 保底值) / (挑战值 - 保底值) × 指标满分
  • 精度：四舍五入保留 1 位小数
  • 边界处理：
  • 保底=挑战 → 除零保护 → 得分=指标满分（或0，按业务定义）
  • 实际值 < 保底值 → 得分=0
  • 实际值 > 挑战值 → 得分=指标满分（不超上限）
  • 实际值=空 → 不参与计算，标记为未完成
  • 持久化：每个指标的得分实时计算并写入 eval_detail.goals_snapshot
定性指标得分
  • 映射表（BR-031）：
  • 远超预期(A) → 115分（区间110-120中值）
  • 超出预期(B) → 99.5分（区间90-109中值）
  • 符合预期(C) → 79.5分（区间70-89中值）
  • 需改进(D) → 59.5分（区间50-69中值）
  • 远低于预期(E) → 24.5分（区间0-49中值）
  • 边界：恰好落在区间边界时，归入该等级（含边界）
  • 映射失败：未选择评价结果时，该指标不参与总分计算
2. 总分计算
总分 = Σ(各指标得分 × 指标权重 / 100) + 加分 - 减分
  • 权重归一化：指标权重以百分比存储（如25表示25%），计算时除以100
  • 加减分范围：0-20分，超出范围校验拒绝
  • 总分截断：上限120分，下限0分
  • 中间结果：每个指标得分实时计算并持久化，总分在直接上级提交时计算
  • 持久化：写入 eval_detail.total_score
3. 总评价结果计算（节点权重）
总评价结果 = Σ(各节点评价结果 × 节点权重) / Σ(节点权重)
  • 节点权重：方案配置时定义（如自评40%+间接上级60%），以百分比存储
  • 归一化：除以权重之和（权重之和必须=100%，不等则阻断保存）
  • 未配置权重：取最后一个填写节点的评价结果（BR-034 降级策略）
  • 部分节点为空：跳过该节点，剩余节点权重重新归一化
  • 持久化：写入 eval_detail.total_eval_result + 映射到等级
4. 总分到等级映射
等级 = grade where grade.min ≤ 总分 < grade.max
  • 边界：总分恰好等于某等级上限，归入该等级（含边界）
  • 超出范围：总分 > 最高等级上限 → 归入最高等级；总分 < 最低等级下限 → 归入最低等级
  • 触发时机：直接上级提交考核时自动计算
  • 持久化：写入 eval_detail.total_eval_result（等级编码）
5. 强制分布实时计算
每次等级调整后实时更新分布计数
  • 触发时机：审定环节每次调整等级后立即重算
  • 计算公式：当前某等级人数 = 该等级已审定人数 / 已审定总人数 × 100%
  • 分母定义（BR-010）：考核组内参与考核的总人数（含跳过审定的员工）
  • 跳过审定的员工：使用方案中配置的固定等级，纳入分母
  • 并发控制：使用数据库行级锁或乐观锁（version字段），防止分布计数不一致
  • 持久化：写入 dist_snapshot 表（实时更新）
逆向场景处理规格
1. 目标制定驳回
  • 触发：间接上级/直接上级点击"不同意"
  • 数据处理（同一条记录）：
  • → goal_detail.status = rejected（未生效）
  • → 数据保留在同一条记录中，员工可修改
  • → 追加审批记录到 approval_history（含驳回原因）
  • 员工重新提交：
  • → 员工修改目标数据（更新同一条记录）
  • → status 回到 pending，重新进入审批流
  • → approval_history 追加新审批记录
  • 目标数据可见性：提交后始终可见，通过 status 标记审批状态
2. 绩效考核驳回（需确认）
  • 当前状态：流程文档03写了间接上级可"驳回到发起人"，但状态机无rejected状态
  • 建议方案A（不支持驳回）：
  • → 绩效考核流程仅支持"提交"，不支持驳回
  • → 间接上级/直接上级只能提交评分
  • → 与状态机定义一致
  • 建议方案B（支持驳回）：
  • → 状态机增加 rejected 状态
  • → 驳回后员工重新填写完成值
  • → 间接上级评分数据保留
  • → 需更新 prd-states.md
  • ⚠️ 需业务确认选择方案A还是B
3. HRBP驳回审定
  • 触发：调整后等级=A时进入HRBP审批，HRBP点击"不同意"
  • 数据处理（追加式）：
  • → 当前 HRBP 行 status=voided（作废）
  • → 流程回到直接上级节点
  • → 强制分布计数回滚（该员工从A等级移除）
  • 直接上级重新审定：
  • → 新增一行 ratification_detail（direct, active）
  • → 重新进入逐级审定流程
  • 历史可追溯：所有 voided 行保留，可查看每次审定的调整原因和结果
  • 对其他员工的影响：不受影响，强制分布计数实时反映当前状态
4. 员工确认不同意
  • 触发：员工点击"不同意"
  • 数据处理：
  • → confirmation.status = pending_hrbp_ack
  • → 通知HRBP查看
  • HRBP知悉后：
  • → confirmation.status = completed
  • → 绩效结果仍锁定（不修改等级）
  • → 线下处理，HRBP手动跟进
  • 系统层面：结果已锁定，不可再次发起确认
  • 如需修正：需发起新的流程实例（终态不可逆）
5. 方案停用时进行中的流程
  • 规则（BR-052）：方案停用不影响已发起的活动
  • 数据快照：
  • → 活动发起时已将方案配置快照到 activity.scheme_snapshot
  • → 方案修改/停用不影响已快照的数据
  • 流程实例：
  • → 已创建的流程实例继续执行
  • → 流程引用的是活动快照中的配置
  • 不可发起新活动：停用方案后，无法基于该方案发起新活动
6. 流程超时自动处理
  • 触发：环节超过配置的 timeout_days 未完成
  • 目标制定超时：
  • → 未填写的员工 → 按"强制结束规则"处理
  • → forced_end_grade = 方案中配置的固定等级
  • → 流程实例标记为 timeout，跳到下一环节
  • 绩效考核超时：
  • → 未评分的节点 → 系统自动以默认评价结果填充
  • → 默认值来源：方案配置的 forced_end_grade
  • → 计算总评价结果时纳入
  • 结果确认超时（BR-005）：
  • → 系统自动确认为"同意"
  • → 记录自动确认标记
7. 并发控制方案
  • 强制分布计数：
  • → 使用 dist_snapshot 表的 version 字段做乐观锁
  • → 更新时校验 version，冲突则重试
  • → 或使用 SELECT ... FOR UPDATE 行级锁
  • 批量评分提交：
  • → 逐条处理（BR-047），非原子操作
  • → 单条失败不影响其他条目
  • → 返回成功/失败明细
  • 流程引擎批量节点：
  • → 为每个员工创建独立的流程实例
  • → 各实例独立流转，互不影响
8. 数据存储策略
  • 方案→活动配置：
  • → HRBP发起活动时，将方案全部配置序列化为JSON
  • → 存入 activity.scheme_snapshot 字段（配置快照，防方案修改影响活动）
  • 目标数据（状态驱动，单条记录）：
  • → 每个员工每个活动只有一条 goal_detail
  • → 提交/驳回/重提都在同一条记录上更新 status
  • → 数据始终可见，通过 status 区分审批状态
  • 考核/审定数据（追加式，每人多行）：
  • → 每个节点人提交 = 新增一行，驳回重提 = 再新增一行
  • → 旧行标记 status=voided（作废），新行 status=active
  • → 查询取 active 最新行 = 当前数据；voided 行 = 历史回溯
  • 面谈→确认（实时引用）：
  • → 确认表单实时读取面谈数据（BR-023 直接引用不复制）
  • → 面谈修改后确认表单同步变化
  • 审批历史：
  • → 目标：approval_history JSON 数组追加
  • → 考核/审定：每行自带 submitter_id + submitted_at，天然形成历史链
