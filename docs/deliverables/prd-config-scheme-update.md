## 1. 配置层级关系

### 1.1 三层配置架构

```
┌─────────────────────────────────────────────────────────┐
│  V1 基础配置层（全局共享，COE维护）                          │
│  绩效周期 / 单位 / 等级规则 / 计算规则 / 衡量标准            │
│  指标库 / 评价规则 / 强制分布规则                           │
└─────────────────────────────────────────────────────────┘
                          ↓ 引用（只读）
┌─────────────────────────────────────────────────────────┐
│  V2 方案配置层（模板，COE维护）                              │
│  绩效方案 → 考核组 / 考核环节 / 流程模板 / 表单模板           │
└─────────────────────────────────────────────────────────┘
                          ↓ 实例化 copy（快照）
┌─────────────────────────────────────────────────────────┐
│  V4 活动层（实例，HRBP发起）                                │
│  考核活动 → 方案快照 + 活动级覆盖配置                        │
└─────────────────────────────────────────────────────────┘
```

### 1.2 配置引用关系

| 上游配置 | 下游引用 | 引用方式 | 可变性 |
|----------|----------|----------|--------|
| 基础配置（V1） | 方案配置（V2） | FK 引用 | 方案引用时读取最新值 |
| 方案配置（V2） | 活动实例（V4） | 快照 copy | 活动启动后不可变 |
| 基础配置（V1） | 活动实例（V4） | 快照 copy | 活动启动后不可变 |

## 2. 基础配置更新规则

### 2.1 可更新配置项

| 配置项 | 更新影响 | 更新时机限制 |
|--------|----------|--------------|
| 绩效周期 | 影响新活动的时间范围 | 活动未进行中可更新 |
| 单位 | 影响新活动的考核单位 | 活动未进行中可更新 |
| 等级规则 | 影响新活动的等级映射 | 活动未进行中可更新 |
| 计算规则 | 影响新活动的得分计算 | 活动未进行中可更新 |
| 衡量标准 | 影响新活动的目标值 | 活动未进行中可更新 |
| 指标库 | 影响新活动的指标选择 | 活动未进行中可更新 |
| 评价规则 | 影响新活动的评价标准 | 活动未进行中可更新 |
| 强制分布规则 | 影响新活动的分布要求 | 活动未进行中可更新 |

### 2.2 更新约束

```yaml
规则 BR-100: 基础配置更新不影响已发起活动
  描述: 已发起的活动使用快照配置，基础配置更新不影响已进行的活动
  场景: COE修改等级规则后，已进行的活动仍使用原等级规则
  实现: 活动启动时生成 scheme_snapshot，包含所有配置快照

规则 BR-101: 基础配置更新影响方案引用
  描述: 方案配置中引用基础配置的部分，读取时取最新值
  场景: COE修改计算规则后，未启用的方案配置显示最新计算规则
  实现: 方案配置页面实时读取基础配置最新值

规则 BR-102: 基础配置删除保护
  描述: 被方案引用的基础配置不可删除
  场景: 某个计算规则被方案A引用，删除时提示"该配置被方案A引用，不可删除"
  实现: 删除前检查引用关系
```

### 2.3 配置更新流程

```
COE 修改基础配置
      ↓
系统检查是否有进行中的活动引用该配置
      ↓
  ┌─ 有进行中活动 → 提示"该配置被进行中活动引用，修改不影响已进行活动，是否继续？"
  │                    ↓ 确认
  │              更新基础配置
  │                    ↓
  │              方案配置页面读取最新配置
  │
  └─ 无进行中活动 → 直接更新基础配置
                          ↓
                  方案配置页面读取最新配置
```

## 3. 方案版本管理

### 3.1 版本号规则

```yaml
规则 BR-110: 方案版本自动递增
  描述: 方案每次保存时版本号自动+1
  场景: COE修改方案配置后保存，版本号从1变为2
  实现: perf_scheme.version 字段自动递增

规则 BR-111: 方案复制重置版本
  描述: 方案复制时版本号重置为1
  场景: COE复制方案A生成方案B，方案B版本号为1
  实现: 复制接口重置version=1

规则 BR-112: 方案启用不改变版本
  描述: 方案启用/停用操作不改变版本号
  场景: 方案A版本3启用，版本号仍为3
  实现: toggle接口不修改version字段
```

### 3.2 版本历史记录

| 字段 | 类型 | 说明 |
|------|------|------|
| `scheme_id` | bigint | 方案ID |
| `version` | int | 版本号 |
| `snapshot` | JSON | 配置快照 |
| `operator_id` | bigint | 操作人 |
| `created_at` | timestamp | 创建时间 |
| `change_log` | text | 变更说明 |

### 3.3 版本查询

```yaml
GET /api/schemes/:id/versions
描述: 查询方案版本历史
响应: 版本列表，包含版本号、快照、操作人、时间、变更说明

GET /api/schemes/:id/versions/:version
描述: 查询指定版本的配置快照
响应: 完整配置快照（考核组、环节、流程模板等）
```

## 4. 活动快照机制

### 4.1 快照生成时机

```yaml
规则 BR-120: 活动启动时生成快照
  描述: HRBP发起活动时，从方案 copy 全部配置生成快照
  场景: HRBP基于方案A发起活动，活动获得方案A的完整配置快照
  实现: perf_activity.scheme_snapshot 字段存储JSON快照

规则 BR-121: 快照包含完整配置
  描述: 快照包含方案的所有配置项（方案有周期类型但不绑定具体周期）
  内容:
    - 方案基本信息（名称、周期类型、等级规则）
    - 考核组配置（规则、排除人员、评价规则）

    - 节点权重配置
```

### 4.3 快照使用规则

```yaml
规则 BR-122: 活动使用快照配置执行
  描述: 活动执行过程中使用快照配置，不读取方案最新配置
  场景: 活动进行中，COE修改方案配置，活动仍使用原快照
  实现: 活动相关查询读取 scheme_snapshot，不读取 perf_scheme 表

规则 BR-123: 活动级配置覆盖优先
  描述: 活动可修改继承的配置，最终按活动的配置执行
  场景: HRBP发起活动后，修改了某个环节的开启方式
  实现: perf_activity 中存储活动级覆盖配置，优先级高于快照

规则 BR-124: 快照不可变
  描述: 活动启动后，scheme_snapshot 字段不可修改
  场景: 任何操作都不能修改已生成的快照
  实现: 数据库层面禁止更新 scheme_snapshot 字段
```

### 4.4 活动绑定绩效周期

```yaml
规则 BR-125: 活动绑定绩效周期
  描述: 活动发起时选择绩效周期，周期信息快照到活动
  场景: HRBP发起活动，选择方案+周期+人员
  实现: perf_activity 新增 cycle_id（FK）+ cycle_snapshot（JSON）
  说明: 方案不含周期，活动单独绑定。一个方案可跨多个周期复用

规则 BR-126: 活动周期快照不可变
  描述: 活动启动后，cycle_snapshot 字段不可修改
  场景: 绩效周期修改/停用不影响已发起活动
  实现: 数据库层面禁止更新 cycle_snapshot 字段

规则 BR-127: 绩效周期删除保护
  描述: 被活动引用的绩效周期不可删除
  检查逻辑:
    1. 查询 perf_activity 表，检查是否有活动引用该周期
    2. 若有引用，返回错误"该周期被活动XXX引用，不可删除"
    3. 若无引用，允许删除

规则 BR-128: 绩效周期修改保护
  描述: 被活动引用的绩效周期不可修改名称/类型/起止月份
  场景: 活动已发起，COE尝试修改周期名称
  结果: 提示"该周期被活动引用，不可修改"
  实现: 更新前检查引用关系
```

### 5.3 变更影响说明

```yaml
规则 BR-130: 草稿方案可自由修改
  描述: 草稿状态的方案可任意修改配置
  场景: COE创建方案后，可反复修改考核组、环节配置
  实现: draft 状态方案无修改限制

规则 BR-131: 已启用方案修改影响新活动
  描述: 已启用方案修改后，新发起的活动使用最新配置
  场景: 方案A启用后，COE修改了考核组配置，HRBP新发起的活动使用新配置
  实现: 新活动启动时读取方案最新配置生成快照

规则 BR-132: 已启用方案修改不影响进行中活动
  描述: 已启用方案修改后，进行中的活动仍使用原快照
  场景: 方案A启用后，COE修改了流程模板，进行中活动仍使用原流程模板
  实现: 活动执行读取 scheme_snapshot，不读取方案最新配置
```

## 6. 方案停用/启用规则

### 6.1 停用规则

```yaml
规则 BR-140: 方案停用条件
  描述: 方案停用后不可发起新活动，已有活动独立运行
  场景: 方案A有3个进行中活动，COE停用方案A
  结果:
    - 方案A状态变为 disabled
    - 3个进行中活动继续执行，不受影响
    - 不可基于方案A发起新活动
  实现: perf_scheme.status = disabled

规则 BR-141: 停用方案不可删除
  描述: 停用状态的方案不可直接删除
  场景: COE停用方案A后，尝试删除
  结果: 提示"方案已停用，不可删除，请先删除关联活动"
  实现: delete 接口检查方案状态

规则 BR-142: 停用方案可重新启用
  描述: 停用的方案可重新启用
  场景: COE停用方案A后，需要重新使用
  结果: 方案A状态变为 enabled，可发起新活动
  实现: toggle 接口切换状态
```

### 6.2 启用规则

```yaml
规则 BR-143: 方案启用校验
  描述: 方案启用时校验必填配置
  校验项:
    - 方案名称不能为空
    - 至少配置1个考核组
    - 至少配置1个考核环节
    - 引用的配置项（等级规则、流程模板等）必须有效
  实现: toggle 接口执行校验

规则 BR-144: 方案启用不影响已有配置
  描述: 方案启用不改变已有配置
  场景: 方案A草稿状态配置了考核组B，启用后配置不变
  实现: 启用操作只修改 status 字段
```

### 6.3 停用流程

```
COE 点击停用方案
      ↓
系统检查是否有进行中的活动
      ↓
  ┌─ 有进行中活动 → 提示"该方案有N个进行中活动，停用后不影响已有活动，是否继续？"
  │                    ↓ 确认
  │              方案状态变为 disabled
  │                    ↓
  │              不可发起新活动，已有活动独立运行
  │
  └─ 无进行中活动 → 方案状态变为 disabled
                          ↓
                  不可发起新活动
```

## 7. 配置继承与覆盖规则

### 7.1 继承规则

```yaml
规则 BR-150: 活动继承方案全部配置
  描述: 活动启动时从方案 copy 全部配置（方案有周期类型但不绑定具体周期，活动单独绑定具体周期）
  继承内容:
    - 方案基本信息（含周期类型）
    - 考核组配置
    - 考核环节配置（开启/结束方式、流程模板、表单模板）
    - 强制分布配置
    - 通知配置
    - 节点权重配置
  实现: 活动启动时生成 scheme_snapshot

规则 BR-151: 活动继承基础配置快照
  描述: 活动启动时 copy 基础配置快照
  继承内容:
    - 等级规则（等级定义、分数区间）
    - 计算规则（公式配置）
    - 衡量标准（目标值、挑战值、保底值）
    - 评价规则（评价等级定义）
    - 强制分布规则（分布比例）
  实现: scheme_snapshot 中包含基础配置快照
```

### 7.2 覆盖规则

```yaml
规则 BR-152: 活动级配置覆盖优先
  描述: 活动可修改继承的配置，最终按活动的配置执行
  可覆盖配置:
    - 环节开启/结束方式
    - 环节时间范围
    - 强制分布参数（分布比例、控制模式）
    - 节点权重
    - 通知配置
  不可覆盖配置:
    - 考核组规则（人员范围）
    - 流程模板
    - 表单模板
    - 等级规则
    - 计算规则
  实现: perf_activity 中存储活动级覆盖配置
```

### 7.3 配置优先级

```
活动级覆盖配置 > 方案快照配置 > 基础配置快照
```

### 7.4 配置查询逻辑

```yaml
查询活动配置时:
  1. 首先检查活动级覆盖配置
  2. 若存在，使用覆盖配置
  3. 若不存在，使用方案快照配置
  4. 方案快照配置中引用的基础配置，使用快照中的值（非最新值）
```

## 8. 异常场景处理

### 8.1 方案配置异常

```yaml
异常 BR-160: 方案引用的配置被删除
  描述: 方案引用的等级规则/流程模板等被删除
  处理:
    - 草稿方案: 提示"引用的配置已被删除，请重新选择"
    - 已启用方案: 不可删除被引用的配置（删除保护）
  实现: 删除前检查引用关系

异常 BR-161: 方案引用的配置被停用
  描述: 方案引用的等级规则/流程模板等被停用
  处理:
    - 草稿方案: 提示"引用的配置已停用，请重新选择"
    - 已启用方案: 不影响，活动使用快照配置
  实现: 草稿方案显示警告，已启用方案忽略
```

### 8.2 活动配置异常

```yaml
异常 BR-162: 活动快照配置失效
  描述: 活动快照中引用的配置在活动进行中被删除
  处理: 活动继续使用快照配置，不读取最新配置
  实现: 活动执行只读 scheme_snapshot，不读取源配置表

异常 BR-163: 活动级覆盖配置冲突
  描述: 活动级覆盖配置与快照配置冲突
  处理: 活动级覆盖配置优先
  实现: 查询配置时优先使用活动级覆盖
```

## 10. 数据库变更

### 10.1 新增表

```sql
-- 方案版本历史表
CREATE TABLE perf_scheme_version (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  scheme_id BIGINT NOT NULL,
  version INT NOT NULL,
  snapshot JSON NOT NULL,
  operator_id BIGINT NOT NULL,
  change_log TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (scheme_id) REFERENCES perf_scheme(id),
  UNIQUE KEY uk_scheme_version (scheme_id, version)
);

-- 活动配置覆盖表
CREATE TABLE perf_activity_config_override (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  activity_id BIGINT NOT NULL,
  config_type VARCHAR(50) NOT NULL,
  config_key VARCHAR(100) NOT NULL,
  config_value JSON NOT NULL,
  operator_id BIGINT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (activity_id) REFERENCES perf_activity(id),
  UNIQUE KEY uk_activity_config (activity_id, config_type, config_key)
);
```

| `perf_activity` | `scheme_version` | 新增 | 方案版本号（快照时的版本） |
| `perf_activity` | `cycle_id` | 新增 | FK → perf_cycle，绑定具体绩效周期 |
