## 1. 活动生命周期

### 1.1 活动阶段

```
┌─────────────────────────────────────────────────────────────────┐
│  活动生命周期                                                     │
│                                                                   │
│  草稿(draft) → 进行中(active) → 已完成(completed) / 已终止(terminated)  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 阶段说明

| 阶段 | 描述 | 可执行操作 |
|------|------|------------|
| `draft` | 草稿状态，未启动 | 编辑、删除、启动 |
| `active` | 进行中，各环节执行中 | 监控、干预、终止 |
| `completed` | 所有环节完成 | 查看、导出 |
| `terminated` | 手动终止 | 查看、导出 |

## 2. 活动状态机

### 2.1 活动主状态图

```
                    ┌──────────────┐
                    │    draft     │
                    └──────┬───────┘
                           │ start
                           ▼
                    ┌──────────────┐
                    │    active    │
                    └──────┬───────┘
                  ┌────────┴────────┐
             complete           terminate
                  ▼                  ▼
           ┌──────────────┐  ┌──────────────┐
           │  completed   │  │  terminated  │
           └──────────────┘  └──────────────┘
```

### 2.2 状态转换表

| 来源 | 事件 | 目标 | 动作 |
|------|------|------|------|
| `draft` | `start` | `active` | 生成参与者、初始化环节状态、分配待办 |
| `active` | `complete` | `completed` | 所有环节完成，写入最终结果 |
| `active` | `terminate` | `terminated` | 手动终止活动 |
| `draft` | `delete` | — | 删除草稿活动 |

### 2.3 活动环节状态

每个参与者在每个环节有独立的状态：

```
┌─────────────────────────────────────────────────────────────────┐
│  环节状态（per participant）                                       │
│                                                                   │
│  not_started → in_progress → completed / skipped / timeout       │
└─────────────────────────────────────────────────────────────────┘
```

| 状态 | 描述 | 触发条件 |
|------|------|----------|
| `not_started` | 未开始 | 活动启动，但该环节未开启 |
| `in_progress` | 进行中 | 环节开启，待办已分配 |
| `completed` | 已完成 | 处理人提交表单 |
| `skipped` | 已跳过 | 环节配置为跳过或审批人为空 |
| `timeout` | 已超时 | 超过配置时间未处理 |

## 3. 活动创建流程

### 3.1 发起活动流程

```
HRBP 发起活动
      ↓
选择绩效方案（方案不含周期，纯模板）
      ↓
选择绩效周期（从预置周期列表选择）
      ↓
系统读取方案配置 + 周期信息
      ↓
生成 scheme_snapshot（方案快照）+ cycle_snapshot（周期快照）
      ↓
设置活动基本信息
  - 活动名称
  - 活动时间（开始/结束日期）
  - 参与人员范围
      ↓
配置活动级覆盖（可选）
  - 环节开启/结束方式
  - 强制分布参数
  - 节点权重
      ↓
预览活动配置
      ↓
确认发起
      ↓
活动状态变为 active
      ↓
系统执行初始化
  1. 生成参与者列表
  2. 初始化各环节状态
  3. 按配置开启第一个环节
  4. 分配待办任务
```

**说明**：方案是纯模板（考核组/环节/流程/表单/权重），不含周期。活动单独绑定绩效周期，一个方案可跨多个周期复用。

### 3.2 参与者生成规则

```yaml
规则 BR-200: 参与者按考核组规则匹配
  描述: 根据方案配置的考核组规则，匹配符合条件的员工
  逻辑:
    1. 遍历考核组（按优先级排序）
    2. 检查员工是否匹配考核组条件（部门/职位/职级）
    3. 检查员工是否在排除人员列表中
    4. 员工匹配第一个命中的考核组
    5. 未匹配任何考核组的员工不参与考核

规则 BR-201: 参与者去重
  描述: 同一员工只能参与一次活动
  场景: 员工同时匹配多个考核组条件
  处理: 按考核组优先级，员工分配到第一个命中的考核组

规则 BR-202: 动态考核参与者
  描述: 开启动态考核时，参与者在各环节开始时重新计算
  场景: 活动进行中，有新员工入职或调岗
  处理: 每个环节开始时，重新匹配参与者
```

## 4. 活动执行流程

### 4.1 环节开启规则

```yaml
规则 BR-210: 自动开启
  描述: 环节配置为自动开启时，按时间或前置环节自动触发
  触发条件:
    - 时间触发: 达到配置的开始时间
    - 前置触发: 前置环节全部完成

规则 BR-211: 手动开启
  描述: 环节配置为手动开启时，需HRBP手动触发
  操作: HRBP在活动管理页面点击"开启环节"

规则 BR-212: 环节依赖
  描述: 环节开启依赖前置环节完成
  依赖关系:
    - 绩效考核 → 依赖目标制定完成
    - 结果审定 → 依赖绩效考核完成
    - 结果确认 → 依赖结果审定完成（月度考核）
    - 绩效面谈 → 独立（年度考核场景）
```

### 4.2 环节执行流程

```
环节开启
      ↓
为参与者创建流程实例
      ↓
分配待办任务
      ↓
参与者处理
      ↓
  ┌─ 提交 → 校验 → 存储 → 推进流程
  │
  ├─ 驳回 → 通知发起人 → 重新提交
  │
  └─ 超时 → 系统自动处理
      ↓
环节完成
      ↓
触发下一环节
```

### 4.3 待办任务分配

```yaml
规则 BR-220: 待办任务分配逻辑
  描述: 环节开启后，为参与者分配待办任务
  分配规则:
    - 目标制定: 分配给员工本人
    - 绩效考核（自评）: 分配给员工本人
    - 绩效考核（间接上级）: 分配给间接上级
    - 绩效考核（直接上级）: 分配给直接上级
    - 结果审定: 分配给直接上级
    - 结果确认: 分配给员工本人
    - 绩效面谈: 分配给员工本人

规则 BR-221: 待办任务状态
  描述: 待办任务有独立的状态
  状态:
    - pending: 待处理
    - in_progress: 处理中
    - completed: 已完成
    - timeout: 已超时
```

## 5. 环节处理逻辑

### 5.1 目标制定环节

```yaml
处理流程:
  1. 员工填写目标（≥2个目标，总权重=100%）
  2. 提交 → 校验 → 存储 goal_detail
  3. 生成间接上级审批待办
  4. 间接上级审批（可跳过）
  5. 直接上级审批（可跳过）
  6. 审批通过 → 目标数据写入考核表单

驳回处理:
  - 间接/直接上级驳回 → 员工重新提交
  - 驳回后，之前被驳回的行 status=voided
  - 员工提交新行，上级重新审批
```

### 5.2 绩效考核环节

```yaml
处理流程:
  1. 员工自评（填写完成值+评价结果）
  2. 提交 → 计算指标得分 → 存储 eval_detail
  3. 间接上级评分（可跳过）
  4. 直接上级评分（计算总分+等级映射）
  5. 考核完成 → 触发结果审定

驳回处理（已确认方案B）:
  - 间接上级驳回 → 该员工所有 active 行 voided
  - 员工修改完成值 → 重新提交 → 全流程重来
  - 原因: 完成值变了，旧评分失效

计算逻辑:
  - 指标得分: calc_rule.formula
  - 总分: Σ(指标得分 × 权重 / 100) + 加分 - 减分
  - 等级: 总分映射到 grade_rule
  - 总评价结果: 按节点权重加权平均
```

### 5.3 结果审定环节

```yaml
处理流程:
  1. 加载考核数据（只读）
  2. 直接上级评级（调整等级）
  3. 间接上级评级（可跳过）
  4. 逐级上级评级（汇报线/组织架构模式）
  5. HRBP审批（仅远超预期A时触发）
  6. 审定完成 → 触发结果确认

强制分布处理:
  - 提交时校验强制分布
  - 强控: 超限阻断提交
  - 弱控: 超限允许提交，附带警告
  - 分母: 考核组参与考核的总人数
  - 分子: 该等级当前 active 行数

驳回处理:
  - HRBP驳回 → 当前行 voided + 强制分布计数回滚
  - 回到直接上级 → 直接上级重新 submit
```

### 5.4 结果确认环节

```yaml
处理流程:
  1. 员工查看确认数据（审定结果+面谈结果+考核结果）
  2. 员工选择同意或不同意
  3. 同意 → 结果锁定，流程结束
  4. 不同意 → HRBP知悉待办
  5. HRBP知悉 → 流程结束，结果锁定

超时处理:
  - 超过配置时间未操作 → 系统自动确认为"同意"
  - 超时自动确认调用 submit(action=agree, operator=system)

关键规则:
  - 员工不同意 → 不驳回，不重新确认，线下处理
  - HRBP角色仅为"知悉"，不审批不驳回
```

### 5.5 绩效面谈环节

```yaml
处理流程:
  1. 员工填写面谈记录
     - 面谈类型（必填）
     - 亮点（≥3项）
     - 不足（≥3项）
     - 下阶段规划（必填）
     - 需求（≥1项）
     - 附件（可选）
  2. 提交 → 校验 → 存储 interview_detail
  3. 上级审批
  4. 上级同意 → 面谈完成
  5. 上级驳回 → 员工重新填写

驳回处理:
  - 上级驳回 → 员工提交新行，旧行 voided
  - 上级重新审批 → ref_detail_id 指向新行
```

## 7. 活动异常处理

### 7.1 异常场景

| 异常场景 | 处理方式 | 影响 |
|----------|----------|------|
| 参与者离职 | 跳过该参与者 | 不影响其他参与者 |
| 参与者调岗 | 按原考核组继续 | 不重新分配考核组 |
| 流程超时 | 按配置的强制结束规则处理 | 使用 forced_end_grade |
| 方案停用 | 活动继续执行 | 不受影响 |
| 流程引擎异常 | 重试或人工干预 | 活动暂停 |

### 7.2 超时处理规则

```yaml
规则 BR-240: 目标制定超时
  描述: 超过配置时间未提交目标
  处理: 使用配置的 forced_end_grade（如C）
  影响: 目标状态变为 timeout

规则 BR-241: 绩效考核超时
  描述: 超过配置时间未完成评分
  处理: 使用配置的 forced_end_grade
  影响: 考核状态变为 timeout

规则 BR-242: 结果审定超时
  描述: 超过配置时间未完成审定
  处理: 使用配置的 forced_end_grade
  影响: 审定状态变为 timeout

规则 BR-243: 结果确认超时
  描述: 超过配置时间未确认
  处理: 系统自动确认为"同意"
  影响: 确认状态变为 completed

规则 BR-244: 绩效面谈超时
  描述: 超过配置时间未完成面谈
  处理: 使用配置的 forced_end_grade
  影响: 面谈状态变为 timeout
```

### 7.3 异常恢复

```yaml
规则 BR-250: 活动暂停恢复
  描述: 活动因异常暂停后可恢复
  恢复条件: 异常原因消除
  恢复操作: HRBP点击"恢复活动"

规则 BR-251: 参与者异常恢复
  描述: 参与者异常处理后可恢复参与
  恢复条件: 参与者状态恢复正常
  恢复操作: 系统自动检测并恢复
```
