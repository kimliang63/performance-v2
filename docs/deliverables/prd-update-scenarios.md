## 1. 配置更新场景

### 1.1 基础配置更新（V1）

#### 场景1：等级规则更新

```
当前状态:
  - 方案A（草稿）引用等级规则X
  - 方案B（已启用）引用等级规则X
  - 活动C（进行中）使用等级规则X的快照

COE修改等级规则X为Y:
  ┌─ 方案A（草稿）: 立即读取等级规则Y
  ├─ 方案B（已启用）: 
  │   ├─ 无进行中活动: 立即读取等级规则Y
  │   └─ 有进行中活动: 新活动读取Y，已进行活动使用X快照
  └─ 活动C（进行中）: 继续使用等级规则X快照
```

**处理规则**：
| 场景 | 处理方式 |
|------|----------|
| 草稿方案 | 立即生效，读取最新配置 |
| 已启用方案（无进行中活动） | 立即生效，新活动使用最新配置 |
| 已启用方案（有进行中活动） | 新活动使用最新配置，已进行活动使用快照 |
| 进行中活动 | 使用快照配置，不受影响 |

#### 场景2：计算规则更新

```
当前状态:
  - 方案A（已启用）引用计算规则X
  - 活动B（进行中）使用计算规则X的快照

COE修改计算规则X的公式:
  ┌─ 方案A（已启用）: 
  │   ├─ 未开始环节: 使用新公式
  │   └─ 已开始环节: 使用原快照公式
  └─ 活动B（进行中）: 
      ├─ 未开始环节: 使用新公式（若方案已更新）
      └─ 已开始环节: 使用原快照公式
```

**处理规则**：
| 环节状态 | 处理方式 |
|----------|----------|
| 未开始（not_started） | 使用最新配置 |
| 进行中（in_progress） | 使用原快照配置 |
| 已完成（completed） | 使用原快照配置 |

#### 场景3：指标库更新

```
当前状态:
  - 方案A（已启用）引用指标KPI_001
  - 活动B（进行中）使用指标KPI_001的快照

COE修改指标KPI_001的衡量标准:
  ┌─ 方案A（已启用）: 
  │   ├─ 未开始环节: 使用新衡量标准
  │   └─ 已开始环节: 使用原快照衡量标准
  └─ 活动B（进行中）: 
      ├─ 未开始环节: 使用新衡量标准（若方案已更新）
      └─ 已开始环节: 使用原快照衡量标准
```

### 1.2 配置删除保护

```yaml
规则 BR-170: 被引用配置不可删除
  描述: 被方案或强制分布规则引用的基础配置不可删除
  检查逻辑:
    1. 查询 perf_scheme 表，检查是否有方案引用该配置
    2. 查询 perf_dist_rule 表，检查是否有强制分布规则引用该配置（等级规则专用）
    3. 若有引用，返回错误"该配置被XXX引用，不可删除"
    4. 若无引用，允许删除

规则 BR-171: 被活动引用配置不可删除
  描述: 被活动快照引用的基础配置不可删除
  检查逻辑:
    1. 查询 perf_activity.scheme_snapshot，检查是否有活动快照引用该配置
    2. 若有引用，返回错误"该配置被活动XXX引用，不可删除"
    3. 若无引用，允许删除
```

### 2.2 已启用方案更新

#### 场景1：方案启用后修改考核组

```
当前状态:
  - 方案A（已启用）配置考核组G1
  - 活动B（进行中）使用考核组G1快照

COE修改方案A的考核组为G2:
  ┌─ 方案A（已启用）: 考核组变为G2
  ├─ 活动B（进行中）: 继续使用考核组G1快照
  └─ 新活动C: 使用考核组G2
```

**处理规则**：
| 活动状态 | 处理方式 |
|----------|----------|
| 进行中活动 | 使用原快照配置，不受影响 |
| 新发起活动 | 使用最新方案配置 |

#### 场景2：方案启用后修改环节配置

```
当前状态:
  - 方案A（已启用）配置环节开启方式为"自动"
  - 活动B（进行中）使用原快照配置

COE修改方案A的环节开启方式为"手动":
  ┌─ 方案A（已启用）: 环节开启方式变为"手动"
  ├─ 活动B（进行中）: 
  │   ├─ 未开始环节: 使用原快照配置（自动）
  │   └─ 已开始环节: 使用原快照配置（自动）
  └─ 新活动C: 使用最新配置（手动）
```

**处理规则**：
| 环节状态 | 处理方式 |
|----------|----------|
| 进行中活动的未开始环节 | 使用原快照配置 |
| 进行中活动的已开始环节 | 使用原快照配置 |
| 新活动的所有环节 | 使用最新方案配置 |

#### 场景3：方案启用后修改流程模板

```
当前状态:
  - 方案A（已启用）引用流程模板F1
  - 活动B（进行中）使用流程模板F1快照

COE修改方案A的流程模板为F2:
  ┌─ 方案A（已启用）: 流程模板变为F2

  │   └─ 已开始环节: 使用原快照流程模板（F1）
  └─ 新活动C: 使用最新流程模板（F2）
```

**处理规则**：
| 场景 | 处理方式 |
|------|----------|
| 进行中活动 | 使用原快照流程模板，不可修改 |
| 新活动 | 使用最新流程模板 |

### 2.3 方案停用/启用更新

#### 场景1：方案停用

```
COE 停用方案A:
  ┌─ 方案A状态: enabled → disabled
  ├─ 进行中活动: 继续执行，不受影响
  └─ 新活动: 不可基于方案A发起
```

**处理规则**：
| 操作 | 处理方式 |
|------|----------|
| 停用方案 | 状态变为disabled，不影响进行中活动 |
| 停用后发起新活动 | 禁止，提示"方案已停用" |
| 停用后删除方案 | 禁止，需先删除关联活动 |

#### 场景2：方案重新启用

```
COE 重新启用方案A:
  ┌─ 方案A状态: disabled → enabled
  ├─ 校验: 必填配置是否完整
  └─ 新活动: 可基于方案A发起
```

**处理规则**：
| 操作 | 处理方式 |
|------|----------|
| 重新启用 | 状态变为enabled，校验必填配置 |
| 重新启用后发起新活动 | 允许，使用最新方案配置 |

### 3.2 活动参与者更新

#### 场景1：动态考核添加参与者

```
当前状态:
  - 活动A（进行中）开启动态考核
  - 目标制定环节已开始

新员工B入职，匹配考核组:
  ┌─ 活动A（进行中）: 
  │   ├─ 目标制定环节: 添加员工B，分配待办
  │   ├─ 绩效考核环节: 添加员工B，等待开启
  │   ├─ 结果审定环节: 添加员工B，等待开启
  │   ├─ 结果确认环节: 添加员工B，等待开启
  │   └─ 绩效面谈环节: 添加员工B，等待开启
  └─ 员工B: 参与活动A的所有环节
```

**处理规则**：
| 场景 | 处理方式 |
|------|----------|
| 动态考核 | 每个环节开始时重新匹配参与者 |
| 新员工入职 | 添加到活动，分配待办 |
| 员工调岗 | 按原考核组继续，不重新分配 |

#### 场景2：移除参与者

```
当前状态:
  - 活动A（进行中）员工C已提交目标

HRBP移除员工C:
  ┌─ 活动A（进行中）: 
  │   ├─ 员工C状态: active → removed
  │   ├─ 目标制定环节: 终止员工C的流程
  │   ├─ 其他环节: 跳过员工C
  │   └─ 强制分布: 从分母中移除
  └─ 员工C: 不再参与活动A
```

**处理规则**：
| 场景 | 处理方式 |
|------|----------|
| 移除参与者 | 终止流程，从分母中移除 |
| 已提交数据 | 保留历史记录，标记为voided |
| 强制分布 | 重新计算分母和分子 |

## 4. 并发更新处理

### 4.1 方案并发更新

```yaml
场景: 两个COE同时修改同一个方案
  - COE1修改考核组配置
  - COE2修改环节配置

处理方式: 乐观锁（version字段）
  1. COE1读取方案，version=1
  2. COE2读取方案，version=1
  3. COE1保存，version检查通过，version=2
  4. COE2保存，version检查失败（期望1，实际2）
  5. 返回错误"方案已被其他人修改，请刷新后重试"
```

**实现逻辑**：
```sql
-- 方案更新时检查版本号
UPDATE perf_scheme 
SET name = ?, version = version + 1, modified_by = ?, updated_at = NOW()
WHERE id = ? AND version = ?;

-- 检查影响行数
IF ROW_COUNT() = 0 THEN
  -- 版本冲突，返回错误
  RETURN ERROR '方案已被其他人修改，请刷新后重试';
END IF;
```

### 4.2 活动并发更新

```yaml
场景: 两个HRBP同时修改同一个活动配置
  - HRBP1修改通知配置
  - HRBP2修改环节时间

处理方式: 乐观锁（updated_at字段）
  1. HRBP1读取活动，updated_at='2026-06-18 10:00:00'
  2. HRBP2读取活动，updated_at='2026-06-18 10:00:00'
  3. HRBP1保存，updated_at检查通过
  4. HRBP2保存，updated_at检查失败
  5. 返回错误"活动配置已被其他人修改，请刷新后重试"
```

**实现逻辑**：
```sql
-- 活动配置更新时检查时间戳
UPDATE perf_activity 
SET name = ?, updated_at = NOW()
WHERE id = ? AND updated_at = ?;

-- 检查影响行数
IF ROW_COUNT() = 0 THEN
  -- 时间戳冲突，返回错误
  RETURN ERROR '活动配置已被其他人修改，请刷新后重试';
END IF;
```

### 4.3 并发提交处理

```yaml
场景: 多个处理人同时提交同一个员工的考核
  - 间接上级A提交评分
  - 直接上级B提交评分

处理方式: 状态机校验
  1. 间接上级A提交，检查状态=pending_indirect_scoring
  2. 直接上级B提交，检查状态=pending_direct_scoring
  3. 若状态不匹配，返回错误"当前状态不允许此操作"
```

**实现逻辑**：
```sql
-- 提交时检查状态
UPDATE eval_detail 
SET status = 'active', submitted_at = NOW()
WHERE activity_id = ? AND employee_id = ? AND status = 'active' AND submitter_role = ?;

-- 检查影响行数
IF ROW_COUNT() = 0 THEN
  -- 状态不匹配，返回错误
  RETURN ERROR '当前状态不允许此操作';
END IF;
```

## 5. 数据一致性保障

### 5.1 快照一致性

```yaml
规则 BR-180: 活动快照生成原子性
  描述: 活动启动时生成快照必须是原子操作
  实现:
    1. 开启事务
    2. 读取方案配置
    3. 生成快照JSON
    4. 创建活动记录（含快照）
    5. 创建参与者记录
    6. 初始化环节状态
    7. 提交事务

规则 BR-181: 快照不可变性
  描述: 活动快照生成后不可修改
  实现:
    1. 数据库层面禁止UPDATE scheme_snapshot字段
    2. 应用层检查快照是否已存在
    3. 若已存在，返回错误"活动快照已生成，不可修改"
```

### 5.2 配置引用一致性

```yaml
规则 BR-182: 配置引用完整性
  描述: 方案引用的配置必须存在且有效
  检查时机:
    - 方案启用时
    - 活动发起时
    - 提交表单时
  检查内容:
    - 等级规则是否存在且状态=enabled
    - 计算规则是否存在且状态=enabled
    - 流程模板是否存在且状态=enabled
    - 表单模板是否存在且状态=enabled
```

### 5.3 强制分布一致性

```yaml
规则 BR-183: 强制分布计数原子性
  描述: 强制分布计数更新必须是原子操作
  实现:
    1. 开启事务
    2. 检查当前分布是否超限
    3. 更新当前计数
    4. 提交事务

规则 BR-184: 强制分布计数回滚
  描述: HRBP驳回时必须回滚强制分布计数
  实现:
    1. 开启事务
    2. 读取被驳回行的adjusted_grade
    3. 更新对应等级的current_count - 1
    4. 标记旧行为voided
    5. 提交事务
```

### 5.4 状态机一致性

```yaml
规则 BR-185: 状态转换原子性
  描述: 状态转换必须是原子操作
  实现:
    1. 开启事务
    2. 检查当前状态是否允许转换
    3. 更新状态
    4. 执行副作用（如创建待办、更新计数等）
    5. 提交事务

规则 BR-186: 状态转换幂等性
  描述: 相同的状态转换重复执行结果一致
  实现:
    1. 检查目标状态是否已到达
    2. 若已到达，直接返回成功
    3. 若未到达，执行状态转换
```
