## 1. 配置变更分类

### 1.1 按配置层级分类

| 配置层级 | 配置项 | 变更影响范围 |
|----------|--------|--------------|
| V1 基础配置 | 绩效周期、单位、等级规则、计算规则、衡量标准、指标库、评价规则、强制分布规则 | 全局影响 |
| V2 方案配置 | 考核组、考核环节、流程模板、表单模板 | 方案级影响 |
| V4 活动配置 | 活动级覆盖配置 | 活动级影响 |

### 1.2 按变更紧急程度分类

| 紧急程度 | 描述 | 审批要求 |
|----------|------|----------|
| 紧急 | 影响进行中活动的配置变更 | 需要审批 + 影响评估 |
| 重要 | 影响已启用方案的配置变更 | 需要审批 |
| 一般 | 影响草稿方案的配置变更 | 无需审批 |

## 3. 配置变更审批流程

### 3.1 基础配置变更审批

```
COE 发起配置变更
      ↓
系统检查变更影响范围
      ↓
  ┌─ 影响已启用方案 → 生成影响评估报告
  │                    ↓
  │              提交审批（COE主管）
  │                    ↓
  │              审批通过 → 执行变更
  │              审批拒绝 → 终止变更
  │
  └─ 仅影响草稿方案 → 直接执行变更
```

### 3.2 方案配置变更审批

```
COE 发起方案变更
      ↓
系统检查方案状态
      ↓
  ┌─ 草稿方案 → 直接执行变更
  │
  └─ 已启用方案 → 检查是否有进行中活动
                    ↓
              ┌─ 有进行中活动 → 生成影响评估报告
              │                    ↓
              │              提交审批（COE主管）
              │                    ↓
              │              审批通过 → 执行变更 + 通知HRBP
              │              审批拒绝 → 终止变更
              │
              └─ 无进行中活动 → 直接执行变更
```

### 3.3 影响评估报告

```yaml
影响评估报告内容:
  1. 变更配置项: 具体修改了什么配置
  2. 影响范围:
     - 影响的方案列表
     - 影响的活动列表
     - 影响的参与者数量
  3. 影响程度:
     - 是否影响进行中活动
     - 是否需要通知HRBP
     - 是否需要回滚方案
  4. 风险评估:
     - 变更可能导致的问题
     - 应对措施
  5. 建议执行时间:
     - 推荐的变更执行时间
     - 是否需要暂停活动
```

### 3.4 审批权限

| 变更类型 | 审批人 | 审批条件 |
|----------|--------|----------|
| 基础配置变更 | COE主管 | 影响评估通过 |
| 方案配置变更 | COE主管 | 影响评估通过 |
| 活动配置变更 | HRBP主管 | 影响评估通过 |
| 紧急配置变更 | COE总监 | 影响评估通过 + 风险评估 |

## 4. 配置变更通知机制

### 4.1 通知场景

| 变更场景 | 通知对象 | 通知内容 |
|----------|----------|----------|
| 基础配置变更 | COE | 配置已更新 |
| 方案配置变更（已启用） | HRBP | 方案配置已更新，新活动使用新配置 |
| 方案配置变更（有进行中活动） | HRBP | 方案配置已更新，进行中活动不受影响 |
| 活动配置变更 | 参与者 | 活动配置已更新 |

### 4.2 通知模板

```yaml
基础配置变更通知:
  标题: 绩效基础配置已更新
  内容: |
    配置项: {配置名称}
    变更内容: {变更说明}
    影响范围: {影响说明}
    生效时间: {生效时间}
  接收人: COE团队

方案配置变更通知:
  标题: 绩效方案配置已更新
  内容: |
    方案名称: {方案名称}
    变更内容: {变更说明}
    影响范围: {影响说明}
    生效时间: {生效时间}
    注意事项: {注意事项}
  接收人: 相关HRBP

活动配置变更通知:
  标题: 考核活动配置已更新
  内容: |
    活动名称: {活动名称}
    变更内容: {变更说明}
    影响范围: {影响说明}
    生效时间: {生效时间}
  接收人: 活动参与者
```

### 4.3 通知渠道

| 渠道 | 适用场景 | 优先级 |
|------|----------|--------|
| 站内信 | 所有变更 | 中 |
| 飞书 | 重要变更 | 高 |
| 邮件 | 正式通知 | 中 |
| 短信 | 紧急通知 | 高 |

## 5. 配置变更回滚机制

### 5.1 回滚场景

| 场景 | 回滚条件 | 回滚方式 |
|------|----------|----------|
| 配置变更错误 | 变更后发现问题 | 回滚到上一版本 |
| 变更影响评估不足 | 变更后发现影响超出预期 | 回滚到上一版本 |
| 审批流程问题 | 审批流程存在漏洞 | 回滚到上一版本 |

### 5.2 回滚流程

```
发现配置变更问题
      ↓
提交回滚申请
      ↓
系统检查回滚可行性
      ↓
  ┌─ 可回滚 → 执行回滚
  │            ↓
  │      通知相关人员
  │
  └─ 不可回滚 → 人工处理
                  ↓
            制定补偿方案
```

### 5.3 回滚规则

```yaml
规则 BR-300: 基础配置回滚
  描述: 基础配置变更后可回滚到上一版本
  回滚条件: 变更后24小时内
  回滚方式: 恢复上一版本配置
  影响: 已使用新配置的方案需要重新生成快照

规则 BR-301: 方案配置回滚
  描述: 方案配置变更后可回滚到上一版本
  回滚条件: 变更后24小时内，且无新活动使用新配置
  回滚方式: 恢复上一版本配置
  影响: 已使用新配置的活动需要重新生成快照

规则 BR-302: 活动配置回滚
  描述: 活动配置变更后可回滚到上一版本
  回滚条件: 变更后24小时内，且当前环节未完成
  回滚方式: 恢复上一版本配置
  影响: 当前环节需要重新执行
```

## 6. 配置变更日志

### 6.1 日志记录

```yaml
配置变更日志内容:
  - 变更ID
  - 变更时间
  - 变更人
  - 变更配置项
  - 变更前值
  - 变更后值
  - 变更原因
  - 影响范围
  - 审批人
  - 审批时间
  - 执行状态
```

### 6.2 日志查询

```yaml
GET /api/config-change-logs
描述: 查询配置变更日志
参数:
  - config_type: 配置类型（基础/方案/活动）
  - config_id: 配置ID
  - operator_id: 操作人
  - start_time: 开始时间
  - end_time: 结束时间
  - page: 页码
  - pageSize: 每页条数
响应: 变更日志列表

GET /api/config-change-logs/:id
描述: 查询指定变更日志详情
响应: 变更日志详情（包含变更前后完整配置）
```

### 6.3 日志保留策略

| 日志类型 | 保留时间 | 说明 |
|----------|----------|------|
| 基础配置变更日志 | 永久 | 重要配置变更永久保留 |
| 方案配置变更日志 | 3年 | 方案配置变更保留3年 |
| 活动配置变更日志 | 1年 | 活动配置变更保留1年 |
