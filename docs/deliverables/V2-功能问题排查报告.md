## 二、整体设计问题（基于三份文档综合理解）

### 2.1 基本信息模块

| # | 问题 | 说明 | 影响 |
|---|------|------|------|
| D-01 | **`cycle_type` 缺失** | 数据库有 `cycle_type`（月度/季度/半年度/年度/自定义），但 PRD 基本信息未提及，原型也无此字段。这是方案的核心属性——决定哪些环节默认开启。 | 无法区分方案类型，模板选择逻辑无法落地 |
| D-02 | **`config_mode` 缺失** | 数据库有 `config_mode`（合并/按考核组），PRD 和原型均未说明：各考核组是共用一套环节配置，还是每组独立配置。 | 向导式配置的结构不明确 |
| D-03 | **"向下公开"字段归属不清** | PRD 基本信息无此字段，原型有，数据库 V1 等级规则有 `is_public`。需确认：方案级别的"向下公开"含义是什么？与等级规则的 `is_public` 是什么关系？ | 字段语义不明 |
| D-04 | **"动态考核"功能边界模糊** | PRD 说"开启时锁定考核关系、动态获取考核关系"，但未说明：动态考核在活动执行时如何工作？参与者是实时计算还是定时快照？ | V4 活动层设计依赖此字段 |

### 2.2 考核组模块

| # | 问题 | 说明 | 影响 |
|---|------|------|------|
| D-05 | **考核组模式定义不完整** | PRD 说"按条件选择/按人员选择"，但未说明：两种模式是互斥的单选，还是可以同时启用（交集）？原型实现为"条件规则+排除人员"的组合。 | 考核组人员确定逻辑不明确 |
| D-06 | **多考核组人员冲突规则缺失** | 员工同时满足多个考核组条件时，按优先级分配到哪个组？PRD 未定义。数据库 `priority` 字段暗示按优先级，但规则未写。 | 人员分配结果不确定 |
| D-07 | **考核组与环节配置的关系不清** | 向导中每个考核组是否独立配置环节？还是所有组共用？PRD 4.2 说"共用环节配置保留"，但 `config_mode` 字段暗示可选。 | 向导交互逻辑不确定 |

### 2.3 考核环节配置模块

| # | 问题 | 说明 | 影响 |
|---|------|------|------|
| D-08 | **PRD 4.6 绩效考核"结束方式"自相矛盾** | PRD 写"选项包含：自动开启、手动开启"，这与"结束方式"语义不符（应该是"定时结束/手动结束"）。疑似复制粘贴错误。 | 开发理解歧义 |
| D-09 | **结束处理规则的"默认评价结果"含义不清** | PRD 4.6 说"结束流程并默认评价结果"，选择"评价等级中的一项"。未说明：这个默认结果是固定值还是按规则计算？ | 超时处理逻辑不明确 |
| D-10 | **强制分布配置不完整** | PRD 4.7 定义了强制分布规则/控制模式/起控人数，但未说明：分布比例在审定过程中如何实时计算？分母是什么？ | 审定环节核心逻辑缺失 |
| D-11 | **审定模式未在方案层定义** | PRD 4.12 说"审定模式在流程配置中定义"，数据库 `prd-states.md` 定义了汇报线/组织架构两种模式。但方案层如何关联此配置？ | 审定流程选择逻辑缺失 |
| D-12 | **"通知"功能范围矛盾** | PRD 4.6/4.7/4.8/4.9 定义了通知配置（类型×渠道），但 PRD 4.12 说"消息通知后续版本统一做"。需确认：V2 是否实现通知？ | 开发范围不确定 |

### 2.4 后端逻辑缺失（原型无法体现）

| # | 问题 | 说明 | 影响 |
|---|------|------|------|
| D-13 | **提交校验规则不完整** | PRD 仅说"校验必填项"，未列出每个环节的具体校验规则。prd-backend-logic.md 定义了部分（目标数≥2、权重=100%），但 PRD 本身未覆盖。 | 前后端校验不一致 |
| D-14 | **追加式存储的版本管理** | prd-backend-logic.md 定义了"每次 submit 新增一行"的追加模式，PRD 未提及。驳回重提时旧行 voided + 新行 active 的规则也仅在后端逻辑文档中。 | PRD 缺少数据存储策略说明 |
| D-15 | **强制分布实时计算逻辑** | prd-backend-logic.md 定义了"审定每次 submit 后立即重算"，PRD 仅提到"系统预计算"选项。两者的关系未对齐。 | 计算时机不确定 |
| D-16 | **方案版本管理机制** | 数据库有 `perf_scheme_version` 表和触发器，PRD 仅说"版本号系统默认生成"。方案保存时自动+1 的逻辑、版本历史快照的使用场景均未在 PRD 中说明。 | 版本管理行为不明确 |
| D-17 | **活动快照机制** | PRD 4.11 说"活动实例化时 copy 方案全部配置"，但未说明：快照是 JSON 整体存储还是逐字段 copy？活动修改配置后如何与快照对比？ | 数据一致性风险 |
