## 是什么
HROne 绩效管理系统的系统设计文档 + 纯 HTML 原型项目

## 技术栈

## 系统架构（核心）

### 三层配置架构
```
V1 基础配置层（COE维护，预置常见周期）
  绩效周期 / 等级规则 / 计算规则 / 衡量标准 / 指标库 / 评价规则 / 强制分布（绑定等级规则）
      ↓ FK 引用（只读）
V2 方案配置层（COE维护，方案有周期类型但不绑定具体周期）
  绩效方案（cycle_type） / 考核组 / 环节配置 / 强制分布参数
      ↓ 快照 copy（不可变）+ 绑定具体周期
V4 活动执行层（HRBP操作）
  考核活动（scheme_snapshot + cycle_snapshot）/ 参与者 / 环节状态 / 操作日志
      ↓ 审定模式/停止人员由流程引擎管理
```

### 五个考核环节
1. **目标制定** → 员工填目标 → 间接上级审批 → 直接上级审批 → 目标生效
2. **绩效考核** → 员工自评 → 间接上级评分 → 直接上级评分（算总分+等级）
3. **结果审定** → 直接上级评级 → 间接上级评级 → 逐级上级评级 → HRBP审批(仅A级)
4. **结果确认** → 员工确认(同意/不同意) → HRBP知悉(仅不同意时)
5. **绩效面谈** → 员工填面谈记录 → 上级审批

### 核心设计决策（已确认）

#### 1. 提交模型
- **无草稿接口**：只有 submit，无 save。本地可暂存（localStorage）
- **一个原子操作**：submit = A校验 → B计算 → C存储 → D提交流程
- **前端纯展示**：不含业务逻辑，所有校验/计算/存储在后端完成
- **AI Agent 兼容**：Agent 调用同一个 submit 接口，传原始输入即可

#### 2. 驳回规则
- **所有流程除发起人外都能驳回**
- 目标制定：间接/直接上级可驳回到员工
- 绩效考核：间接/直接上级可驳回到员工（全部 active 行 voided，全流程重来）
- 结果审定：间接/逐级上级可驳回到直接上级；HRBP可驳回到直接上级
- 绩效面谈：上级可驳回到员工
- 结果确认：**不支持驳回**，"不同意"走 HRBP 知悉分支，终态不可逆

#### 3. 计算逻辑（两层级公式）

**层级1：单表单总分**
```
定性指标得分 = eval_rule.grades[选择的等级].score（由评价规则配置，不硬编码）
定量指标得分 = calc_rule.formula(actual_value, grades)
单表单总分 = Σ(指标得分 × 指标权重 / 100) + 加分 - 减分
```

**层级2：最终得分**
```
最终得分 = Σ(各节点表单总分 × 方案配置的节点权重 / 100)
等级 = grade_rule[最终得分]（等级仅用于展示和强制分布，不参与计算）
```

#### 4. 数据存储模式
- **追加式**（目标/考核/审定/面谈）：每次提交新增一行，驳回重提再新增一行，旧行 voided
- **状态驱动**（确认环节）：每人一行，终态不可逆
- **快照不可变**：活动发起时方案配置序列化为 JSON，后续不受方案修改影响

#### 5. 方案与周期的关系
- 方案有 `cycle_type`（月度/季度/半年度/年度/自定义），是周期类型分类，不绑定具体周期
- 活动发起时选择方案+具体周期（cycle_id），周期快照到活动（cycle_snapshot）
- 一个方案可跨多个同类型周期复用（如"标准季度考核方案"用于 Q1/Q2/Q3）
- 绩效周期被活动引用时不可修改/删除（报表按周期汇总，修改会导致历史数据关联断裂）
- 绩效周期预置常见类型，可新建但不建议修改已有周期

#### 6. 强制分布
- **绑定等级规则**：强制分布规则必须关联等级规则（grade_rule_id），分布比例按等级定义
- **分母**：考核组参与考核的总人数（含跳过审定+含未审定）
- **分子**：该等级当前 active 行数（需去重：一个员工只算一次）
- **并发控制**：乐观锁（version 字段）或行级锁
- **强控**：超限阻断提交；**弱控**：超限允许提交附带警告
- **等级规则变更联动**：COE 增删等级时，引用该等级规则的强制分布规则需同步更新

#### 6. 配置变更影响
- 草稿方案：立即生效
- 已启用方案（无进行中活动）：立即生效
- 已启用方案（有进行中活动）：新活动生效，已进行活动使用快照
- 进行中活动：使用快照，不受影响

## 开发规则
- 一套代码越滚越大，不按版本分目录
- 版本确认后快照到 `confirmed/vN/`，后续不改
- 改造已有功能必须在 CHANGELOG.md 记录
- Git 不是必须的，目录结构即版本控制
- **废弃所有旧 React 原型**，全部重新输出纯 HTML 原型

- 流程引擎平台：流程设计器、流程管理、表单管理、消息中心
