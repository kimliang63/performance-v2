# 绩效V2 需求分析与差距识别（更新版）

## P0 问题已解决

### 1. 考核组模式
- 默认1个考核组
- 用户可创建多个，每个设置优先级和流入规则

### 2. 动态考核
- 锁定时：创建活动时将当前人员+考核流程节点处理人一并存入活动
- 不随组织架构变动更新
- 手动加入考核人时按加入时刻锁定

### 3. 绩效流程操作
- 增加：驳回、撤回、催办（原仅有提交、转交）

### 4. 绩效表单字段（已从电子表格获取）

| 顺序 | 节点名称 | 节点类型 | 流入规则 | 处理人规则 |
|------|----------|----------|----------|-----------|
| 1 | 目标制定 | 处理 | 开启目标制定 | 被考核人 |

## V2+V3 合并输出建议

**结论：建议合并。**

理由：
1. V2配置的方案→V3的活动执行是一条完整链路
2. 表单字段跨版本：目标制定(V2)→考核(V2)→审定(V2)→确认(V2)→面谈(V3)，数据流贯通
3. 流程引擎V3能力（顺序审批、移动端）直接被绩效流程消费

合并方式：一套原型代码，页面分区
- 配置区：绩效方案配置（V2）
- 执行区：绩效活动+各环节表单（V2/V3共用）
- 流程引擎组件：设计器、待办中心、审批详情（V2/V3共用）
